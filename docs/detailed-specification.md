# 주야간보호 업무 일지 생성 서비스 - 세부 명세서

## 1. 프로젝트 개요 및 원칙
본 프로젝트는 주야간보호 요양보호사가 짧은 사실 메모만으로 공식적인 어조의 일일/주간 관찰 기록 초안을 자동 생성할 수 있도록 돕는 웹 애플리케이션입니다.

### 1.1. 핵심 설계 원칙
- **미니멀리즘**: 불필요한 화려함이나 'AI Slop'으로 느껴질 수 있는 과장된 UI를 철저히 배제하고, 핵심 기록 작업에 집중할 수 있는 깔끔한 인터페이스를 유지합니다.
- **점진적 확장**: 최초 릴리스에는 반드시 필요한 핵심 흐름(대상자 선택 -> 메모 -> 확인 -> 초안 생성)만을 단일 흐름으로 구축하며, 부가적인 페이지 생성을 지양합니다.
- **SSOT (Single Source of Truth)**: 모든 코드(상수, 타입, 컴포넌트 로직) 및 문서는 중복을 최소화하여 유지보수성을 극대화합니다. 단일 책임 원칙을 준수하여 파일이 비대해지면 분리합니다.

---

## 2. 디자인 및 UI/UX 명세

### 2.1. 디자인 시스템
- **최소한의 페이지 이동**: 페이지 전환을 최소화하고, 다이얼로그나 바텀 시트 등을 활용하여 맥락이 끊기지 않는 비동기 UI를 제공합니다.
- **접근성 및 사용성**: 주 사용자(IT에 익숙하지 않은 요양보호사)를 고려하여 버튼 영역을 충분히 확보하고 명확한 텍스트 위주로 구성합니다.
- **친근한 형태 (라운드 처리)**: 딱딱한 IT 기기나 사무용 소프트웨어 같은 느낌을 피하고 사용자에게 따뜻하고 친근한 인상을 주기 위해, 버튼, 카드, 다이얼로그 등 주요 UI 요소의 모서리를 충분히 둥글게(Round) 처리합니다.

### 2.2. 시맨틱 색상 시스템
- **하드코딩 절대 금지**: 임의의 헥스 코드(`ex: text-[#333333]`)나 테일윈드 기본 컬러 중 프로젝트 스케일에 없는 색상 사용을 금지합니다.
- **의미론적(Semantic) 토큰 사용**: `primary`, `surface`, `text`, `border`, `status-success`, `status-danger` 등의 역할 기반 이름을 부여합니다.
- **스케일 기반 조절**: 선언된 색상 내에서만 `100~900` 형태의 스케일로 명도를 조절하여 사용합니다. 필요한 경우 `tints.dev`를 활용해 일관된 스케일을 추출합니다.
- **클래스명 사용 컨벤션**: `tailwind.config.ts` 설정을 통해 `text-primary-400`, `text-primary-600`, `bg-primary-500` 등의 방식으로 직관적으로 사용할 수 있도록 세팅해야 합니다.

## 3. 기술 스택 명세

### 3.1. 프레임워크 및 패키지 매니저
- **프레임워크**: Next.js (App Router) 기반의 서버 사이드 렌더링 및 API 라우트 통합 관리
- **패키지 매니저**: `pnpm` (속도 및 디스크 공간 최적화, 의존성 관리)
- **언어**: TypeScript를 통한 정적 타이핑으로 안정성 확보

### 3.2. 상태 관리 및 데이터 페칭
- **우선 순위**: Next.js와 React의 내장 기능(Server Components, Server Actions, `use` 훅 등)을 1순위로 사용하여 페이지 전환 없는 비동기 요청을 처리합니다.
- **Tanstack Query 도입 기준**: 서비스 초기 규모에서는 Next.js 내장 기능만으로 충분할 것으로 평가됩니다. 추후 클라이언트 측에서의 복잡한 캐시 무효화나 낙관적 업데이트(Optimistic Updates)가 빈번해지는 시점에 제한적으로 도입합니다.

### 3.3. 데이터베이스 및 ORM
- **데이터베이스**: `SQLite` (1인용 로컬/MVP 환경에 최적화)
- **ORM**: `Drizzle ORM` (가볍고 타입 안정성이 뛰어난 SQL 빌더 역할)

### 3.4. 스타일링 및 UI 라이브러리
- **Tailwind CSS**: 전역 스타일링 및 유틸리티 클래스 적용. (`tailwind.config.ts`에 시맨틱 컬러 강제 적용)
- **Shadcn-ui 도입 전면 재평가**:
  - **평가 결론**: **도입하지 않음 (배제)**
  - **이유**: Shadcn-ui가 기본적으로 제공하는 보더 반경(border-radius), 밀도 높은 여백, 회색조(grayscale) 중심의 스타일링은 전형적인 'SaaS 대시보드' 특유의 차가운 느낌을 주어, 본 프로젝트가 지향하는 사용자 친화적이고 미니멀한 방향성과 맞지 않습니다. 기본 제공되는 테마나 CSS 변수 구조 역시 오버엔지니어링될 소지가 큽니다.
  - **방침**: 전형적인 SaaS 형태의 UI를 탈피하기 위해 Shadcn-ui가 생성하는 사전 정의된 컴포넌트는 사용하지 않습니다. 모든 UI는 프로젝트만의 시맨틱 색상 및 여백 스케일을 바탕으로 처음부터 작성하며, `Dialog`나 `Select`와 같이 접근성(a11y)과 키보드 내비게이션 처리가 필수적인 컴포넌트 개발 시에만 아무런 스타일이 없는 **Base UI**를 직접 가져와 우리만의 스타일을 입혀 사용합니다.

## 4. 개발 컨벤션

### 4.1. 선언형 프로그래밍
- '어떻게' 수행할 것인지(절차적, 명령형 DOM 조작)를 지양하고, 상태에 따라 화면이 '무엇'을 보여줄 것인지 선언적으로 작성합니다.
- 복잡한 로직은 순수 함수로 분리하여 테스트 가능하게 만듭니다.

### 4.2. 리액트 컴파일러 최적화 대응
- 향후 React 컴파일러 도입을 전제로 하여, 메모이제이션을 위한 불필요한 `useCallback`, `useMemo` 사용을 지양합니다.
- 컴포넌트의 본질적인 렌더링 흐름과 비즈니스 로직에만 집중하여 가독성을 높입니다.

### 4.3. 파일 및 책임 분리
- 하나의 파일(또는 컴포넌트)이 데이터 패칭, 비즈니스 로직, 뷰 렌더링의 책임을 모두 가지지 않도록 합니다.
- 뷰 컴포넌트, 커스텀 훅(로직), 유틸리티 함수, 타입 정의 파일 등으로 책임을 엄격히 분리하여 파일 길이를 짧게 유지합니다.

## 5. 최소 구현 핵심 기능 (MVP 빌드업)

1. **메인 화면 (홈)**
   - 대상자 리스트 표시.
   - **대상자 등록**: '추가' 버튼을 눌러 대상자 이름 입력 후 확인하여 즉시 리스트에 반영.
2. **대상자 상세 화면 (기록 작업)**
   - **주간 기록 내역 조회**: 해당 주간에 이미 입력된 과거 내역들을 보여줌.
   - **오늘 입력 내역 (일일 기록)**: 당일 내역이 비었으면 입력을 유도함. **(핵심 가치)** 요양보호사는 부담 없이 단편적인 문장이나 여러 개의 키워드만 툭툭 입력할 뿐이며, 서비스(AI)가 이를 정제하여 공식적인 포맷의 **"인지 관련 요양보호기록 문장"**과 **"행동 관련 요양보호기록 문장"**으로 완벽하게 나누어 작성해주는 것이 사용자가 얻는 가장 큰 결과물입니다.
   - **주간 리포트 발간**: 한 대상자에게 한 주에 2일 이상 일일 기록이 입력된 경우, '주간 리포트 발간' 버튼 노출 및 활성화.
   - **주간 기록 생성 및 저장**: 버튼 클릭 시, 해당 주의 기록들을 바탕으로 **인지와 행동이 종합된 주간 요양보호기록**을 발간하고 내용을 저장함.
