# 개발 컨벤션

이 문서는 개발 중 판단이 필요한 순간에 함께 참고할 기준을 정리한다. 모든 상황을
규칙으로 만들고 복잡한 개발 이론을 적용하는 것이 아닌, 코드가 단순하고 예측 가능한 방향으로 유지하여 
어떠한 개발자도 빠르게 적응할 수 있는 환경을 만드는 것이 목표로 한다.

제품 문구와 기록 생성 원칙은 [제품 기준](./product.md), 기술 선택과 코드 경계는
[아키텍처](./architecture.md), 실행과 검증 방법은 [운영 가이드](./operations.md)를
따른다.

## 공통 원칙

### 하나의 사실은 한곳에서 관리한다

같은 값을 여러 상태나 파일에 중복해서 관리하지 않는다. 다른 값으로부터 계산할 수
있다면 별도 상태로 저장하지 않고 필요할 때 계산한다.

코드, 문서, 설정도 같은 원칙을 따른다. 패키지 버전은 `package.json`, 데이터
모델은 `src/db/schema.ts`처럼 이미 기준이 있는 정보는 다른 문서에 복제하지 않고
링크로 연결한다.

### 책임이 커지면 나눈다

하나의 파일이 화면 렌더링, 데이터 조회, 변환, 외부 통신 등 여러 책임을 동시에
떠안아 이해하기 어려워지면 역할에 따라 분리한다.

반대로 재사용 가능성을 예상해서 처음부터 작은 파일과 추상화를 많이 만들지는
않는다. 실제 책임의 경계나 반복이 드러났을 때 나눈다.

### 도메인을 중심으로 확장한다

현재 구조를 억지로 전면 개편하지 않는다. 기능이 커질수록 관련 타입, 쿼리, 상태,
UI를 `src/features/<domain>` 가까이에 모으는 방향으로 점진적으로 이동한다.

여러 도메인에서 실제로 공유하는 UI와 로직만 `src/components`, `src/lib` 같은
공통 영역에 둔다.

## React

### App Router의 기본 흐름을 따른다

NextJS의 철학에 맞지 않는 무분별하게 `'use client'`를 도배하는 안티 패턴을 피하고
컴포넌트는 Server Component를 기본으로 한다. 상태, 이벤트, 브라우저 API처럼
클라이언트 실행이 필요한 부분에만 `'use client'`를 사용하고 그 범위를 작게
유지한다.

서버에서 처리할 수 있는 데이터 조회와 변환을 불필요하게 클라이언트로 옮기지
않는다.

### 선언적인 코드를 우선한다

화면이 현재 상태에서 어떻게 보여야 하는지를 표현한다. DOM을 직접 조작하거나
effect로 상태를 맞추는 명령형 흐름은 브라우저 API나 외부 시스템 연동처럼 필요한
경우에만 사용한다.

props나 기존 상태에서 계산할 수 있는 값은 렌더링 중 계산한다. 이를 다시 state로
복사하고 effect로 동기화하지 않는다.

### 불필요한 수동 최적화를 피한다

React Compiler 기반 코드를 작성한다. `useMemo`,
`useCallback`, `memo`를 관성적으로 추가하는 레거시 리액트의 안티 패턴을 절대 적용하지 않는다.


## 상태 관리

상태는 필요한 범위와 수명에 맞는 가장 단순한 위치에 둔다.

- 서버에서 가져와 화면을 구성하는 데이터는 Server Component를 기본으로
  한다.
- 검색 조건, 선택한 주처럼 URL로 공유하거나 새로고침 후 유지할 값은 `nuqs`로
  관리한다.
- 무한 스크롤, polling, 클라이언트 캐시 동기화가 필요한 서버 데이터에만 TanStack
  Query를 사용한다.
- 입력 중인 폼 값이나 열림 상태처럼 컴포넌트에만 필요한 값은 로컬 state로 둔다.

동일한 데이터를 URL, Query cache, 로컬 state에 동시에 원본처럼 보관하지 않는다.
여러 표현이 필요하다면 기준이 되는 상태를 하나 정하고 나머지는 그 값에서
**계산**한다.

## 스타일
- 색상은 `src/app/globals.css`에 정의된 의미 기반 디자인 토큰을 사용한다. 컴포넌트에
hex, rgb 같은 색상 값을 직접 작성하거나 같은 의미의 색상 토큰을 반복해서 만들지
않는다.
- 새로운 색상 토큰은 특정 화면의 장식이 아니라 여러 화면에서 공유할 의미가 있을 때
추가한다. 
- 다양한 색상의 사용은 최소화하고 기존 색상에서 scale하는 방식을 적용한다. 기존 시맨틱 컬러에서 tints.dev API를 활용하여 컬러 스케일 값을 확보한다. 
- 개발 시에는 모바일 화면을 우선하고, 주요 조작 요소는 충분한 터치 영역을 확보한다.
- 모달과 토스트처럼 접근성이 필요한 복합 UI는 Base UI primitive를 우선 사용한다.

## 프로젝트 규칙

- 사용자 날짜는 `YYYY-MM-DD` 형식으로 다루고 오늘과 주간 계산은
  `src/lib/dateUtils.ts`의 KST 기준 함수를 사용한다.
- DB 접근은 Drizzle ORM을 사용하며 데이터 모델 변경은 `src/db/schema.ts`에서
  시작한다.
- 관찰 메모, 생성 결과, API 키 등 민감정보를 서버 로그에 남기지 않는다.
- 사용자에게 보이는 안내와 오류는 이해하기 쉬운 한국어로 작성한다.
- AI가 만든 기록은 저장 전에 사용자가 확인하고 수정할 수 있어야 한다.

## 인증 및 보안 컨벤션

- **환경변수 검증**: `.env`의 환경변수는 직접 `process.env`로 접근하지 않고, `src/env.ts`에서 Zod로 유효성을 검증한 후 `env` 객체를 통해 접근한다.
- **세션 및 인가(Auth) 확인**: API 라우트 및 Server Action에서의 권한 검증은 `src/lib/auth.ts`의 `getSession()` 헬퍼를 공통으로 사용하여 DRY 원칙을 지킨다.
- **라우팅 보호**: 페이지 접근 권한(ex: 로그인 유저의 로그인 페이지 접근 차단 등)은 Next.js 16 컨벤션에 맞춰 `src/proxy.ts`에서 중앙 집중적으로 처리한다.

## 검증

커밋 시 Husky와 lint-staged가 스테이징된 JavaScript와 TypeScript 파일에
`eslint --fix`를 실행한다.

변경 후에는 `pnpm lint`와 `pnpm exec tsc --noEmit`을 실행한다. 라우팅, 서버 경계,
빌드 설정 또는 배포 동작에 영향을 주는 변경은 `pnpm build`까지 확인한다.

현재 별도 포매터와 자동화된 테스트 체계는 없다. 도구를 도입하면 실행 명령과
적용 범위를 `package.json` 및 관련 문서에 함께 반영한다.
